# v0.30 to v0.31

## Overview

v0.31 introduces enroute delay strategy, splitting delay tracking into enroute and terminal components. This required renaming several configuration keys to reflect that distinction.

There are three breaking changes that require updates to your `Maestro.yaml`.

## Before You Start

Back up your existing `Maestro.yaml` before making any changes.

## Breaking Changes

### `Trajectories` renamed to `TerminalTrajectories`

The `Trajectories` key under each airport has been renamed to `TerminalTrajectories` to distinguish it from the new `EnrouteTrajectories` section.

Before:

```yaml
Airports:
  - Identifier: YSSY
    Trajectories:
      - FeederFix: RIVET
        ...
```

After:

```yaml
Airports:
  - Identifier: YSSY
    TerminalTrajectories:
      - FeederFix: RIVET
        ...
```

**Steps:** Rename every `Trajectories:` key under each airport to `TerminalTrajectories:`.

---

### `DefaultTTGMinutes` renamed to `DefaultTimeToGoMinutes`

The `DefaultTTGMinutes` key under each airport has been renamed to `DefaultTimeToGoMinutes`.

Before:

```yaml
Airports:
  - Identifier: YSSY
    DefaultTTGMinutes: 18
```

After:

```yaml
Airports:
  - Identifier: YSSY
    DefaultTimeToGoMinutes: 18
```

**Steps:** Rename every `DefaultTTGMinutes:` key under each airport to `DefaultTimeToGoMinutes:`.

---

### `ControlAction` colour source split into `RequiredControlAction` and `RemainingControlAction`

The `ControlAction` colour source used in label item `ColourSources` has been replaced by two separate values: `RequiredControlAction` and `RemainingControlAction`. These correspond to the control action for the required delay and the remaining delay respectively.

Before:

```yaml
Labels:
  Layouts:
    - Identifier: Enroute
      Items:
        - {Type: RequiredDelay, ColourSources: [ControlAction]}
        - {Type: RemainingDelay, ColourSources: [ControlAction]}
```

After:

```yaml
Labels:
  Layouts:
    - Identifier: Enroute
      Items:
        - {Type: RequiredDelay, ColourSources: [RequiredControlAction]}
        - {Type: RemainingDelay, ColourSources: [RemainingControlAction]}
```

**Steps:** In each label layout item that uses `ControlAction` in `ColourSources`:

- On `RequiredDelay` items, replace `ControlAction` with `RequiredControlAction`.
- On `RemainingDelay` items, replace `ControlAction` with `RemainingControlAction`.

---

## New Features

These additions are optional. Existing configurations will continue to work without them.

### Delay component selection for `RequiredDelay` and `RemainingDelay`

`RequiredDelay` and `RemainingDelay` label items now accept a `Component` property that controls which portion of the split delay is displayed.

| Value | Description |
|-------|-------------|
| `Total` | Sum of enroute and terminal delay (default) |
| `Enroute` | Enroute delay only |
| `Tma` | Terminal delay only |

```yaml
- {Type: RequiredDelay, Width: 3, Component: Enroute, ColourSources: [RequiredControlAction]}
- {Type: RemainingDelay, Width: 3, Component: Total,  ColourSources: [RemainingControlAction]}
```

If `Component` is omitted, it defaults to `Total`.

### Enroute trajectories

Each airport can now define an `EnrouteTrajectories` section that specifies how much delay can be absorbed in the enroute phase for flights arriving via each entry point. Flights with no matching entry use `DefaultMaxEnrouteLinearDelayMinutes` (default: `8`).

```yaml
Airports:
  - Identifier: YSSY
    DefaultMaxEnrouteLinearDelayMinutes: 8

    EnrouteTrajectories:
      - EntryPoint: VELGI
        FeederFix: RIVET
        MaxEnrouteLinearDelayMinutes: 8
        ShortcutTimeToGainMinutes: 3
      - EntryPoint: NONUP
        FeederFix: RIVET
        MaxEnrouteLinearDelayMinutes: 6
```

See [Plugin Configuration](../plugin-configuration#enroute-trajectories) for full property descriptions.
